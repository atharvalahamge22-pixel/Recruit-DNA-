## What this is

RecruitDNA is an AI candidate-ranking engine. Given a job description and a pool of
candidate profiles, it produces a ranked shortlist — but instead of ranking by keyword
or embedding similarity alone, it scores each candidate across **six dimensions** inspired
by the biological relationship between **genotype** (raw data) and **phenotype**
(observed, real-world expression):

| Dimension | Weight | What it measures |
|---|---|---|
| Skill Match | 25% | Semantic overlap between JD requirements and candidate skills |
| Learning Velocity | 25% | Is role complexity increasing over the candidate's career? |
| Curiosity Index | 20% | GitHub activity, research papers, hackathons, blog posts |
| Adaptability | 15% | Successful domain/role pivots |
| Stability | 10% | Tenure patterns and growth continuity |
| Communication | 5% | Resume clarity and quantified achievements |

These combine into a single **Professional Phenotype™ score (0–100)**, and every
candidate's rank ships with a plain-English, fully traceable explanation.

## Why this approach

Standard candidate-matching systems (keyword search, or even sentence-embedding
cosine similarity) answer: *"how closely does this resume match the JD?"*

RecruitDNA answers a different question: *"how likely is this candidate to actually
succeed and grow in this role?"* A candidate with a perfect keyword match but zero
growth signals may rank below a candidate with a slightly lower skill overlap but
strong learning velocity and curiosity — because the second candidate's trajectory
predicts stronger long-term performance.

## Quick start

```bash
git clone <this-repo>
cd RecruitDNA
pip install -r requirements.txt
python main.py
```

This runs the full pipeline against the bundled sample data
(`data/job_description.json` + `data/candidates.csv`) and writes the ranked
output to `output/ranked_candidates.csv`.

### Using your own data

```bash
python main.py --jd path/to/job_description.json --candidates path/to/candidates.csv --output path/to/output.csv
```

Your `candidates.csv` needs these columns (see `data/candidates.csv` for the exact format):

```
candidate_id, name, years_experience, current_role, skills, education,
past_roles_count, avg_tenure_months, role_complexity_trend, github_repos,
oss_contributions, research_papers, hackathon_wins, blog_posts,
domain_switches, successful_pivot, resume_word_count,
quantified_achievements, certifications
```

Your `job_description.json` needs this shape (see `data/job_description.json`):

```json
{
  "title": "...",
  "seniority": "...",
  "required_skills": ["...", "..."],
  "preferred_skills": ["...", "..."],
  "domain_context": "...",
  "implicit_signals": "..."
}
```

## How explainability works (and why it's hallucination-safe)

The `xai_explanation` column is **never** freely generated by an LLM. Every
sentence is template-filled from the candidate's already-computed scores and
raw input fields. For example:

> "Exceptional Learning Velocity (94/100) — role complexity trend: increasing.
> Also strong Curiosity Index (88/100) — 6 GitHub repos, 3 papers, 2 hackathon
> wins. No significant weak signals detected across all six dimensions."

Every number and claim in that sentence traces directly back to a value in
`candidates.csv`. There is no step where a model invents justification text —
this eliminates the hallucination risk that comes with free-form LLM explanations.

## Performance

- Scores 20 candidates against a JD in **under 0.2 seconds** on CPU
- No GPU required
- Skill-match scoring uses Sentence-Transformers when available (better semantic
  matching, e.g. "Deep Learning" ↔ "Neural Networks"), and automatically falls
  back to TF-IDF cosine similarity if the model can't be downloaded — so the
  pipeline runs in fully offline / restricted-network environments too

## Tech stack

Python 3.11 · pandas · NumPy · scikit-learn · Sentence-Transformers (optional,
with TF-IDF fallback) · FAISS-ready architecture for large-scale candidate pools

## Note on the sample dataset

`data/candidates.csv` contains 20 **synthetic** candidate profiles created to
demonstrate the full pipeline end-to-end. To run against the official hackathon
dataset, replace this file (matching the column schema above) and re-run
`main.py` — no code changes required.
